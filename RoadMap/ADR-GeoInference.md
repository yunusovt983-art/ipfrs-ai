## ADR-001 — Транспорт выкачки блоков: request-response, не bitswap-behaviour (на старте)

**Контекст.** `fetch_block_from_peer` — заглушка (`ipfrs-network/src/node.rs:1315`). Нужен
реальный обмен блоками по swarm. В `IpfrsBehaviour` (`node.rs:288`) bitswap отсутствует;
в `ipfrs-transport` есть готовый движок планирования `BitswapExchange` (`bitswap.rs:75`),
но он не подключён к libp2p.

**Варианты.**
- **A. Полноценный Bitswap как `NetworkBehaviour`** — совместимость с IPFS Bitswap 1.2,
  want-have/want-block, но большой объём и риск.
- **B. libp2p `request-response` протокол `/ipfrs/blockfetch/1.0.0`** — минимальный wire:
  `Request{cid}` → `Response{block | not_found}`. Логику приоритетов/выбора пиров берём из
  уже готового `ipfrs-transport::BitswapExchange`.

**Решение: B сейчас, A потом.** Стартуем с `request-response` — это кратчайший путь к
рабочему P2P GET и MVP гео-инференса. Полный Bitswap-behaviour (A) выносим в отдельную
фазу как оптимизацию (батчинг want-list, дедуп дубликатов, IPFS-совместимость).

**Последствия.**
- (+) Малый, проверяемый объём; разблокирует Фазы 2–4.
- (+) Переиспользуем `BitswapExchange` для планирования (какому пиру слать), а wire — тонкий.
- (−) Нет IPFS-совместимости на первом этапе; нет батч-обмена want-list.
- Миграция на A не ломает API `Node::get` — меняется только транспортный слой.

## ADR-002 — Манифест модели: IPLD DAG-CBOR со списком CID слоёв

**Контекст.** Для «моделей как контента» (Фаза 2) нужна каноничная, контент-адресуемая
структура, описывающая модель как DAG из блоков-слоёв. В коде уже есть аналог —
`KnowledgeBaseIpld { rules: Vec<cid>, ... }` (`ipld_codec.rs:88`) и детерминированный
`build_dag_cbor_cid` (`ipld_codec.rs:123`).

**Варианты.**
- A. Один монолитный блок модели (просто, но нет дедупа слоёв и частичной загрузки).
- B. JSON-манифест вне IPLD (не контент-адресуем каноночно).
- **C. IPLD DAG-CBOR манифест** со ссылками на слои по CID.

**Решение: C.**
```
ModelManifest (DAG-CBOR):
{
  arch:    String,           // "llama-7b", "bert-base"
  dtype:   String,           // "f16" | "int8" | ...
  version: String,           // semver
  layers:  [ LayerRef { cid, name, shape: [u64], offset?: u64 } ],
  metadata: { ... }          // входит в хеш → разные метаданные = разный model_cid
}
```
`model_cid = build_dag_cbor_cid(manifest)`; каждый `LayerRef.cid` — отдельный `Block`
(через `safetensors_support.rs` / `gradient/arrow_ipc.rs`).

**Последствия.**
- (+) Дедуп одинаковых слоёв между версиями; частичная/ленивая загрузка (Фаза 2.2).
- (+) Детерминизм CID (как у правил, `ipld_codec.rs:171`) → реплики совпадают побайтно.
- (+) Естественно ложится на TensorSwap (`want_tensor` тянет нужные `layer_cid`).
- (−) Нужен кодек слоёв (safetensors↔Block) — но он уже частично есть.

## ADR-003 — Маршрутизация: гибрид (Geo latency + Semantic MoE + DHT providers)

**Контекст.** «Куда слать инференс» зависит от трёх осей: (1) кто географически ближе
(латентность), (2) кто держит нужный `model_cid` (доступность), (3) какой пир-эксперт
ближе по эмбеддингу (MoE). В коде есть `GeoRouter`/`PeerSelector`/`QualityPredictor`
(`facade.rs:100-103`, не подключены) и семантический DHT (`dht.rs:148`).

**Варианты.**
- A. Только Kademlia provider records — кто угодно с моделью (без учёта латентности).
- B. Только семантический DHT — близость по эмбеддингу (без учёта гео/доступности).
- **C. Гибрид трёх осей** с приоритетом.

**Решение: C, порядок отбора:**
1. **Кандидаты** = `find_providers(model_cid)` (Kademlia, `node.rs:1214`) — у кого вообще есть модель.
2. **Фильтр/сортировка по гео** = `PeerSelector` + `QualityPredictor` (RTT, регион, загрузка).
3. **MoE-уточнение** (опц., Фаза 5.3) = среди близких — ближайший по эмбеддингу
   (`find_nearest_peers`, `dht.rs:148`).

**Последствия.**
- (+) Латентность как первичный критерий (гео-инференс), но только среди держателей модели.
- (+) MoE включается опционально, не усложняя MVP.
- (−) Требует «оживить» Tier B (Фаза 3) — кормить модули событиями swarm.

## ADR-004 — Хеджированные запросы вместо одиночного выбора

**Контекст.** Между ДЦ хвостовая латентность (p99) высока; один выбранный пир может
«затупить». Want-list Bitswap уже поддерживает несколько пиров (`bitswap.rs:184`).

**Варианты.**
- A. Один лучший пир (просто, но плохой p99).
- **B. Хеджирование**: слать k ближайшим, брать первый CID-проверенный ответ, остальные cancel.

**Решение: B** с настраиваемым `k` (деф. 2–3) и бюджетом времени.

**Последствия.**
- (+) Срезает p99; устойчивость к «медленному» пиру.
- (−) Доп. трафик (×k); смягчается ранним `cancel` после первого валидного ответа.
- Политику выносим в `GeoInferPolicy { k, deadline_ms, verify }` (см. каркас Фазы 4).

## ADR-005 — Доверие к чужому инференсу: proof-carrying + опц. голосование

**Контекст.** В открытой сети пир может вернуть неверный результат. Есть `ProofTree`
(`proof_tree.rs:199`) и `provenance.rs:239`.

**Решение.** Базово — **proof-carrying**: ответ несёт `model_cid + input_cid + result_cid
(+ ProofTree для символики)`; клиент проверяет целостность по CID. Для high-assurance —
опциональное **избыточное исполнение на N пирах + голосование большинством** (BFT-lite),
включается политикой.

**Последствия.** (+) Воспроизводимость/аудит без доверия к пиру. (−) Голосование ×N стоит
ресурсов — только по флагу для критичных запросов.
