# Вклад в upstream (текущий)

> Этот форк основан на Apache 2.0 кодовой базе.  
> **Все исправления ошибок и функции должны быть внесены в исходный проект.**

---

## Философия

Цель - **не конкурентный форк**:

1. **Стабилизировать** IPFRS v0.2.1
2. **Документировать** архитектуру
3. **Расширить** функциями
4. **Внести вклад в upstream**

**Обоим репозиториям выгода.**

---

## Что внести в upstream

### ✅ Всегда внести

- Исправления ошибок (все 6 критических)
- Патчи безопасности
- Улучшения производительности
- Улучшения тестов
- Документация
- Исправления CI/CD

### ⚠️ Обсудить первым

- Новые функции
- Изменения API
- Рефакторинг архитектуры
- Новые зависимости

### ❌ Сохранить локально

- Ваша документация (Wiki_Arch_Claude)
- Руководства развёртывания
- Экспериментальные функции
- Брендирование форка

## Процесс: исправление ошибки

### Шаг 1: Создать выпуск в исходном репозитории

```bash
gh issue create -t "Security: JWT uses MD5 instead of HS256" \
  -b "Описание, где это, почему это проблема"
```

### Шаг 2: Создать feature branch

```bash
git clone https://github.com/cool-japan/ipfrs.git
git checkout -b fix/jwt-hs256-auth
# ... изменить код ...
cargo test --all-features
cargo clippy --all -- -D warnings
```

### Шаг 3: Открыть PR

```bash
git push -u origin fix/jwt-hs256-auth
gh pr create --title "fix: JWT MD5 → HS256 HMAC" --body "Fixes #123"
```

### Шаг 4: Ответить на рецензию

- Внести запрошенные изменения
- Отправить follow-up коммиты

### Шаг 5: Merge

- PR объединён
- Ваш форк cherry-picks исправление

---

## Процесс: новая функция

### Шаг 1: Обсудить в выпуске

```markdown
# RFC: S3 Gateway

Предложение, мотивация, дизайн, оценка затрат, вопросы для мейнтейнеров.
```

### Шаг 2: Получить одобрение

Ждите обратной связи перед началом основной работы.

**Хорошие признаки:**
- "Это ценно"
- "Давайте с MVP"
- "Запланировано для v0.3.0"

### Шаг 3-5: Реализация (как bug fix)
