# Обзор домена IPFRS

**Краткое резюме**: IPFRS — контент-адресуемая распределённая ФС, где **всё
адресуется по содержимому** (`cid = hash(data)`), а сверх классического IPFS
добавлены два домена интеллекта: семантический поиск и нейро-символический вывод.
Эта страница задаёт единый язык, на котором говорят все остальные.

---

## 1. Какую задачу решает домен

Классические файловые системы адресуют данные по **местоположению** (путь, адрес
сервера). IPFRS адресует по **содержимому**: имя блока выводится из его байтов.
Из этого вытекают три фундаментальных свойства, которые пронизывают всю систему:

1. **Дедупликация бесплатна** — одинаковое содержимое → одинаковый CID → хранится
   один раз (`ipfrs-storage/src/blockstore.rs:319` — пропуск записи при наличии CID).
2. **Целостность проверяема** — любой узел может независимо проверить `cid == hash(data)`
   (`ipfrs-core/src/block.rs:117`), не доверяя источнику.
3. **Местоположение неважно** — блок можно взять у любого пира, у которого он есть;
   маршрутизацией занимается DHT (`ipfrs-network`).

Поверх этой основы IPFRS добавляет **«…with ML»**: контент-адресуемые тензоры,
векторный семантический поиск и движки логического/вероятностного вывода.

---

## 2. Ядро домена (Core Domain) и вспомогательные поддомены

В терминах DDD не все части системы одинаково ценны:

| Тип поддомена | Контексты | Почему |
|---------------|-----------|--------|
| **Core Domain** (уникальная ценность) | TensorLogic, Semantic | Именно нейро-символический вывод + семантика отличают IPFRS от обычного IPFS |
| **Supporting** (нужно, но не уникально) | Storage, Network, Transport | Контент-адресуемое хранилище и P2P — необходимая, но известная техника |
| **Generic** (можно купить/взять готовое) | Interface/Gateway | gRPC/GraphQL/WS/HTTP, Auth, TLS — стандартные технологии |

> Это объясняет, почему `ipfrs-tensorlogic` — **самый большой крейт** (≈156K строк,
> 20+ движков вывода): на Core Domain тратится больше всего инженерных усилий.

## 3. Единый язык (Ubiquitous Language)

Термины, которыми обязаны пользоваться код, документация и обсуждения. Сгруппированы
по слою.

### 3.1 Контент-адресация (Shared Kernel)

| Термин | Точное значение в коде | Источник |
|--------|------------------------|----------|
| **CID** | Content Identifier = `(version, codec, multihash)`. Самоописывающийся адрес содержимого. Тип из внешнего крейта `::cid`, расширен трейтом `CidExt`. | `cid.rs:6,381` |
| **Block** | Неизменяемая единица хранения `{cid, data}`, где `cid = hash(data)`. Размер ∈ [1 B, 2 MiB]. Корневой агрегат ядра. | `block.rs:57,77` |
| **Multihash** | Хеш-часть CID: `(код-алгоритма, дайджест)`. Поддержано 8 алгоритмов (SHA-2/3, BLAKE2/3). | `cid.rs:12` |
| **Codec** | Multicodec (`u64`): как трактовать байты блока (`RAW=0x55`, `DAG_CBOR=0x71`…). | `cid.rs:582` |
| **IPLD** | InterPlanetary Linked Data — модель данных из 9 вариантов; канонично кодируется в DAG-CBOR. | `ipld.rs:19` |
| **Link** | `Ipld::Link(Cid)` — типизированная ссылка между блоками. Превращает плоское хранилище в Merkle-DAG. | `ipld.rs:37` |
| **DAG** | Направленный ацикличный граф блоков, связанных CID-ссылками. | `dag.rs`, `chunking.rs:328` |
| **Chunk** | Лист-фрагмент большого файла (по умолч. 256 KiB), каждый → один лист-Block. | `chunking.rs:16` |
| **CAR** | Content Addressable aRchive — переносимый контейнер DAG (в IPFRS — вариант с per-block сжатием, не чистый CARv1). | `car.rs:334` |

### 3.2 Хранение (Storage Context)

| Термин | Значение | Источник |
|--------|----------|----------|
| **BlockStore** | Порт-Репозиторий: `put/get/has/delete/list_cids`. | `traits.rs:9` |
| **Pin** | Маркер корневого множества, защищающий блок от GC. Типы: Direct / Recursive / Indirect. | `pinning.rs:22` |
| **GC** | Mark-and-sweep сборка мусора от корней-пинов. | `gc.rs:163` |
| **Tier** | Класс хранения по «температуре» доступа: Hot / Warm / Cold / (Frozen). | `tiering.rs:32` |

### 3.3 Сеть и обмен (Network + Transport)

| Термин | Значение | Источник |
|--------|----------|----------|
| **PeerId** | Идентичность узла libp2p = `hash(публичный ключ ed25519)`. | `node.rs:506` |
| **Kademlia / DHT** | Распределённая хеш-таблица для маршрутизации контента (k=20, α=3). | `node.rs:655` |
| **Provider record** | «Пир P хранит CID X». Публикуется `start_providing`, ищется `get_providers`. | `node.rs:1198` |
| **Bitswap** | Протокол обмена блоками: want / have / block / dont-have / cancel. | `transport/bitswap.rs:75` |
| **WantList** | Приоритетная очередь CID, которые узел хочет получить. | `want_list.rs:220` |
| **Session** | Агрегат, группирующий связанные запросы блоков с жизненным циклом. | `session.rs:199` |
| **TensorSwap** | Расширение Bitswap для потоковой передачи тензоров (учёт дедлайнов и зависимостей). | `tensorswap/core.rs:57` |

### 3.4 Интеллект (Semantic + TensorLogic) — Core Domain

| Термин | Значение | Источник |
|--------|----------|----------|
| **Embedding** | Плотный `Vec<f32>` (по умолч. 768-мерный), представляющий смысл контента. | `hnsw.rs:92` |
| **HNSW** | Hierarchical Navigable Small World — слоистый граф для приближённого поиска соседей (ANN). | `hnsw.rs:84` |
| **DiskANN / Vamana** | Дисковый ANN-индекс (алгоритм Vamana). | `diskann.rs:592` |
| **Term / Predicate / Rule** | Символический IR: переменная/константа/функция; именованное отношение; хорн-клауза. | `ir.rs:13,165,217` |
| **KnowledgeBase** | Агрегат символики: `{facts, rules}`. | `ir.rs:277` |
| **Inference engine** | Движок вывода (SLD, Tabling, Bayesian, Fuzzy, Epistemic S5, PLN, …; всего 20+). | см. [[07-TensorLogicContext]] |
| **Autograd** | Обратное автодифференцирование (reverse-mode AD). | `autograd.rs:52` |
| **FedAvg** | Федеративное усреднение градиентов между пирами. | `gradient/backward_pass.rs:19` |
| **Neural-symbolic fusion** | Смешение символического вывода и нейросетевого сходства эмбеддингов. | `neural_symbolic.rs:489` |

## 4. Главные инварианты системы

Это правила, нарушение которых ломает корректность. Они enforced в коде (где —
указано); часть «спроектированных» инвариантов на деле не проверяется — это
помечено ⚠️ и детализировано в [[11-RealityCheck]].

| # | Инвариант | Где enforced |
|---|-----------|--------------|
| I1 | **`cid = hash(codec, data)`** при создании блока | `block.rs:70`, `cid.rs:337` |
| I2 | **Размер блока ∈ [1 B, 2 MiB]** | `block.rs:77` (но обходится `from_parts`) |
| I3 | **Каноничность IPLD** (ключи map отсортированы → детерминированный CID) | `ipld.rs:35,339` |
| I4 | **GC никогда не удаляет запиненные/достижимые блоки** | `gc.rs:255`, `ipfrs/src/gc.rs:100` |
| I5 | **Дедуп при записи** (существующий CID не перезаписывается) | `blockstore.rs:319` |
| I6 | **Согласованность размерности эмбеддингов** | `hnsw.rs:159` |
| I7 | **Детерминизм CID-адресуемых правил** (равные правила → равный CID) | `ipld_codec.rs:171` |
| I8 | **Аккумуляция градиентов при fan-out** (корректность обратного прохода) | `autograd.rs:217` |
| ⚠️ I9 | **Проверка целостности при чтении** — *спроектирована, но НЕ выполняется* по умолчанию: `get` использует `from_parts` без `verify()` | `blockstore.rs:350` |

## 5. Как это всё связано (один взгляд)

```
Пользователь добавляет файл
   │
   ▼  chunking → Block(ы), cid = hash(data)         [Shared Kernel]
   ▼  put в BlockStore (дедуп, кэш, bloom, sled)     [Storage]
   ▼  provide(cid) в Kademlia DHT                     [Network]
   │
   ▼  (опционально) index_content → embedding → HNSW  [Semantic]
   ▼  (опционально) add_fact / add_rule → KB (cid)    [TensorLogic]

Другой узел запрашивает cid
   │
   ▼  get: cache → store → DHT find_providers          [Application → Storage → Network]
   ▼  fetch блока у пира (Bitswap/Session)             [Transport]
   ▼  backfill в локальный store, verify               [Storage]
```
