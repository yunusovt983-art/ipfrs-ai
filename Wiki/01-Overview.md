# Обзор IPFRS: Система, которая мыслит

**Краткое резюме**: IPFRS — это распределённая файловая система на Rust, которая объединяет хранилище данных с автоматическим рассуждением. Отличается от обычного IPFS добавлением семантического поиска и логического вывода.

---

## Главный вопрос

> Как сделать так, чтобы распределённое хранилище данных **понимало** смысл данных и могло **автоматически рассуждать** о них?

**Традиционный ответ**: Отделить это на два слоя.
- **Слой 1**: Хранилище (IPFS)
- **Слой 2**: Интеллект (приложение над IPFS)

**Ответ IPFRS**: Объединить в один слой.
- **Интеллект встроен в хранилище**
- Данные + смысл = одна система

---

## Двухслойная архитектура

```
┌───────────────────────────────────────────────┐
│         ЛОГИЧЕСКИЙ СЛОЙ (Мозг)                │
│  • Semantic Router: HNSW (векторный поиск)    │
│  • TensorLogic Store: Логический вывод        │
│  • Knowledge Base: Факты + Правила            │
├───────────────────────────────────────────────┤
│         ФИЗИЧЕСКИЙ СЛОЙ (Тело)                │
│  • Block Storage: Неизменяемые блоки (CID)    │
│  • Network Stack: libp2p + QUIC + DHT         │
│  • Transport: Bitswap для обмена блоками      │
└───────────────────────────────────────────────┘
```

**Ключевая идея**: Эти слои НЕ отделены. Они работают синергично:
- Семантический индекс направляет сетевой маршрут
- Логический вывод оптимизирует размещение блоков
- Распределённый вывод использует возможности peer'ов
- Сетевое обнаружение информирует семантическое индексирование

---

## Пять независимых доменов (DDD)

IPFRS следует **Domain-Driven Design**. Система состоит из пяти orthogonal domains:

| Домен | Язык | Ответственность |
|-------|------|-----------------|
| **Storage** | Block, CID, DAG | "Что у нас есть?" |
| **Network** | Peer, DHT, Multiaddr | "Где находятся узлы?" |
| **Semantic** | Embedding, HNSW | "Что это означает?" |
| **Logic** | Term, Rule, Predicate | "Что мы можем вывести?" |
| **Transport** | Session, WantList, Bitswap | "Как обмениваться надёжно?" |

Каждый домен может развиваться независимо, но все работают вместе через чистые интерфейсы (трейты).

## Кубки данных, или как данные рождаются в IPFRS

### 1️⃣ User добавляет файл

```
User: "Добавь документ.pdf"
         ↓
CLI читает файл → Chunking → 391 блок по 256KB
         ↓
Для каждого блока:
  • Compute CID = hash(data)
  • Сохранить в Storage (Sled)
  • Объявить в Network (DHT)
  • Индексировать в Semantic (если нужно)
  • Сохранить в Logic (если есть структура)
         ↓
User получает: "Добавлено! Root CID = bafybeig..."
```

**Время**: ~900ms для 100MB (в основном сетевая задержка)

### 2️⃣ User ищет похожие документы

```
User: "Найди документы, похожие на эту тему"
         ↓
ML Model: Преобразует текст в вектор [0.1, -0.2, 0.3, ...]
         ↓
HNSW индекс: Поиск k-NN в пространстве 768 измерений
         ↓
Results: Top 10 похожих документов с оценками сходства
         ↓
User видит: Список с заголовками и оценками
```

**Время**: ~1ms (если результат в кеше) до ~10ms (HNSW поиск)

### 3️⃣ User выполняет логический запрос

```
User: "Найди всех предков Alice"
         ↓
Knowledge Base: 
  • Факты: parent(alice, bob), parent(bob, charlie)
  • Правила: ancestor(X,Y) :- parent(X,Y)
             ancestor(X,Z) :- parent(X,Y), ancestor(Y,Z)
         ↓
Inference Engine: Backward chaining
         ↓
Results: [bob, charlie, ...]
```

**Время**: 1-5ms для типичных запросов

## Ключевые инварианты

Эти правила НИКОГДА не должны нарушаться:

### 1. Детерминизм хеширования
```
hash(data) = CID   (всегда)

Если это нарушено → коррупция обнаружена!
```

### 2. Глобальная идентичность
```
Если у двух узлов одинаковый CID → одинаковые данные
```

### 3. Неизменяемость блоков
```
Block.data не меняется после создания.
Если нужно изменить → создай новый блок с новым CID.
```

### 4. Репутация честна
```
reputation_score пропорционален успешному обслуживанию
Плохие peer'ы автоматически избегаются.
```

### 5. Логическая консистентность
```
Правила не должны приводить к противоречиям.
```

## Основные числа (быстрая справка)

```
СКОРОСТЬ:
  Block GET (кеш):     30µs ← очень быстро (оперативная память)
  Block PUT:           50µs ← тоже быстро
  Block GET (диск):   100µs ← медленнее (SSD)
  HNSW поиск:         1-10ms ← зависит от размера индекса
  DHT lookup:       150-300ms ← сетевая задержка
  Логический вывод:   1-5ms ← обычно быстро

ПРОПУСКНАЯ СПОСОБНОСТЬ:
  Block ops:    20-100k ops/sec (в зависимости от параллелизма)
  Queries:      1k semantic searches/sec
  DHT:          100 lookups/sec

ПАМЯТЬ (для 1TB данных):
  LRU Cache:       2.0 GB
  HNSW Index:      1.5 GB
  Peer State:    100 MB
  Sessions:      50 MB
  Metadata:      200 MB
  Runtime:       600 MB
  ────────────
  Итого:       ~4.5 GB RAM
```

**Rust + Tokio + libp2p + Sled + HNSW = Распределённая система, понимающая смысл**

```
Runtime:        Tokio 1.52 (async/await)
Networking:     libp2p 0.56 (QUIC, TCP, WebSocket)
```

## Почему это важно?

### Проблема: Информационная изоляция
Данные хранятся, но система не "понимает" их смысл.

```
Файл PDF о машинном обучении
         ↓
IPFS: "Это просто набор байтов: bafybeig..."
         ↓
Чтобы найти похожие файлы → вручную перебирать или использовать внешний сервис
```

### Решение: Встроенный интеллект
```
Файл о машинном обучении
         ↓
IPFRS: 
  • Понимает: embedding = [0.14, -0.09, 0.23, ...]
  • Автоматически индексирует
  • Может найти похожие файлы за 1ms
  • Может ответить на вопросы: "Что упоминается в этом файле?"
```

## Ключевые идеи (зашифрованные)

1. **Content-addressing**: Все идентифицируется по содержанию, а не по местоположению
2. **Deteminism**: Одни данные → один CID → глобальное согласие
3. **Distribution**: Нет центрального органа; peer-to-peer
4. **Intelligence**: Семантика встроена в архитектуру
